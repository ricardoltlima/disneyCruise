Spring Boot Request Lifecycle

Core interview question: what happens when an HTTP request reaches a Spring Boot backend application?

Client sends GET /api/reservations/123 Load balancer or gateway forwards traffic to the service Embedded server, usually Tomcat, receives the HTTP request Spring MVC DispatcherServlet receives the request Spring maps the request to the correct controller method Controller handles HTTP boundary concerns Service layer executes the business workflow Repository or client accesses data or external systems Response DTO returns through the controller Jackson serializes Java object to JSON

Spring Boot

Spring Boot helps build production-ready Java services by providing auto-configuration, embedded servers, dependency injection, and integrations for web APIs, persistence, security, monitoring, and testing.

Embedded Server

A Spring Boot REST API usually runs with an embedded server such as Tomcat. Instead of deploying a WAR file to an external server, the application can often be started with:

java -jar reservation-service.jar

DispatcherServlet

The DispatcherServlet is the central entry point for Spring MVC requests. It receives the HTTP request, uses handler mappings to find the matching controller method, invokes that method, and coordinates the response.

Controller

The controller owns the HTTP boundary: routes, path variables, query parameters, request bodies, validation trigger, and response status. Controllers should stay thin and delegate business rules to services.

@RestController
@RequestMapping("/api/reservations")
public class ReservationController {
    private final ReservationService reservationService;

    public ReservationController(ReservationService reservationService) {
        this.reservationService = reservationService;
    }

    @GetMapping("/{id}")
    public ReservationResponse getReservation(@PathVariable Long id) {
        return reservationService.getReservation(id);
    }
}

Service

The service layer owns the business workflow. It applies business rules, coordinates repositories and external clients, manages transactions, handles domain-level exceptions, and decides what should happen for a use case.

@Service
public class ReservationService {
    private final ReservationRepository reservationRepository;

    public ReservationService(ReservationRepository reservationRepository) {
        this.reservationRepository = reservationRepository;
    }

    public ReservationResponse getReservation(Long id) {
        Reservation reservation = reservationRepository.findById(id)
                .orElseThrow(() -> new ReservationNotFoundException(id));

        return new ReservationResponse(
                reservation.getId(),
                reservation.getGuestName(),
                reservation.getStatus()
        );
    }
}

Repository

The repository layer abstracts persistence. With Spring Data JPA, the framework can generate implementations for common operations such as findById, save, delete, and findAll.

@Repository
public interface ReservationRepository extends JpaRepository<Reservation, Long> {
}

DTO vs Entity

DTOs define the shape of data crossing the API boundary. Entities represent persisted domain data and are usually mapped to database tables or documents. Avoid exposing entities directly from APIs because that can leak internal database structure and make API contracts harder to evolve.

public record ReservationResponse(
        Long id,
        String guestName,
        String status
) {}

Beans and Dependency Injection

Core interview question: how does Spring create objects and connect them together?

Spring Bean

A Spring bean is an object managed by the Spring container. Spring creates it, injects its dependencies, manages its lifecycle, and can apply framework behavior around it.

Beans can be discovered with annotations such as @Component, @Service, @Repository, @Controller, @RestController, and @Configuration. They can also be declared manually through @Bean methods.

Spring Container

The Spring container, usually represented by ApplicationContext, discovers bean definitions, creates objects, wires dependencies, manages lifecycle callbacks, and makes beans available for injection.

@SpringBootApplication
public class ReservationApplication {
    public static void main(String[] args) {
        SpringApplication.run(ReservationApplication.class, args);
    }
}

Component Scanning

@SpringBootApplication includes component scanning. By default, Spring scans the package where the main application class lives and its subpackages. If a bean is not being injected, one thing to check is whether it is inside the component scan path.

Constructor Injection

Constructor injection is preferred because required dependencies are explicit, fields can be final, objects cannot be created in an incomplete state, and classes are easier to unit test without starting Spring.

@Service
public class ReservationService {
    private final ReservationRepository reservationRepository;

    public ReservationService(ReservationRepository reservationRepository) {
        this.reservationRepository = reservationRepository;
    }
}

Multiple Beans and Qualifier

If multiple beans match the same dependency type, Spring needs disambiguation. Use @Qualifier to name the dependency explicitly or @Primary to define a default bean.

public PaymentService(@Qualifier("stripePaymentClient") PaymentClient paymentClient) {
    this.paymentClient = paymentClient;
}

Bean Scope and Concurrency

Most Spring beans are singleton by default, meaning one instance per Spring container. Since the same service instance can handle many requests concurrently, request-specific mutable data should stay in method parameters or local variables, not service fields.

Avoid this pattern: storing currentReservationId or similar request data as a field in a singleton service. Concurrent requests can overwrite each other's data and cause race conditions.

Configuration Beans

Use @Configuration and @Bean when you need custom construction logic or want to register a third-party object with Spring.

@Configuration
public class AwsConfig {
    @Bean
    public DynamoDbClient dynamoDbClient() {
        return DynamoDbClient.builder()
                .region(Region.US_EAST_1)
                .build();
    }
}

Bean Lifecycle

  1. Bean definition discovered.
  2. Object instantiated.
  3. Dependencies injected.
  4. Initialization callbacks run.
  5. Bean is used by the application.
  6. Shutdown callbacks run when the app stops.

REST API Design

Core interview question: how do you design a clean, reliable REST API in Spring Boot?

Resource-Oriented URLs

REST APIs usually model URLs around resources, not actions. The URL identifies the thing, and the HTTP method expresses the action.

Good

GET /api/reservations/123
POST /api/reservations
PATCH /api/reservations/123
DELETE /api/reservations/123

Less Ideal

GET /api/getReservation?id=123
POST /api/createReservation
POST /api/updateReservation
POST /api/deleteReservation

HTTP Methods

  • GET: read a resource.
  • POST: create a resource or trigger a non-idempotent operation.
  • PUT: replace a resource.
  • PATCH: partially update a resource.
  • DELETE: delete or cancel a resource.

Status Codes

  • 200 OK: successful read or update.
  • 201 Created: resource created, often with a Location header.
  • 202 Accepted: request accepted for asynchronous processing.
  • 204 No Content: successful operation with no response body.
  • 400 Bad Request: invalid request syntax or validation failure.
  • 401 Unauthorized: missing or invalid authentication.
  • 403 Forbidden: authenticated caller is not allowed.
  • 404 Not Found: resource does not exist.
  • 409 Conflict: state conflict or duplicate business operation.
  • 429 Too Many Requests: rate limit exceeded.
  • 500 Internal Server Error: unexpected server failure.
  • 503 Service Unavailable: temporary outage or dependency unavailable.

DTOs

Use DTOs for request and response contracts instead of exposing entities directly. DTOs separate the API contract from the persistence model, prevent leaking internal fields or sensitive data, and let the API evolve without forcing every database change to become a client-facing change.

public record CreateReservationRequest(
        @NotBlank String guestName,
        @NotNull Long cruiseId,
        @Min(1) int partySize
) {}

public record ReservationResponse(
        Long id,
        String guestName,
        Long cruiseId,
        int partySize,
        String status
) {}

Validation

@Valid tells Spring to apply validation rules on the request DTO before the controller method continues. If validation fails, Spring raises a validation exception and the service layer is not called.

@PostMapping
public ResponseEntity<ReservationResponse> createReservation(
        @Valid @RequestBody CreateReservationRequest request
) {
    ReservationResponse response = reservationService.createReservation(request);
    URI location = URI.create("/api/reservations/" + response.id());
    return ResponseEntity.created(location).body(response);
}

Global Exception Handling

@RestControllerAdvice centralizes exception handling across controllers. It keeps controllers clean, maps exceptions to proper HTTP status codes, and returns consistent error responses.

@RestControllerAdvice
public class ApiExceptionHandler {
    @ExceptionHandler(ReservationNotFoundException.class)
    public ResponseEntity<ApiError> handleNotFound(ReservationNotFoundException ex) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
                .body(new ApiError("RESERVATION_NOT_FOUND", ex.getMessage()));
    }
}

Error Response Design

Reliable APIs should return predictable errors with stable error codes, clear messages, and trace IDs for supportability.

{
  "code": "VALIDATION_ERROR",
  "message": "Request validation failed",
  "traceId": "abc-123",
  "fieldErrors": [
    {
      "field": "partySize",
      "message": "partySize must be at least 1"
    }
  ]
}

Pagination and Versioning

Avoid unbounded list endpoints. Use page and size for simple cases, and cursor-based pagination for high-volume or frequently changing datasets.

GET /api/reservations?page=0&size=50
GET /api/reservations?limit=50&cursor=eyJpZCI6MTIzfQ

Prefer backward-compatible API changes when possible. For breaking changes, use a clear versioning strategy such as /api/v1/reservations and a migration plan.

Reliability Roadmap

A mature backend call is not just one service calling another. It includes validation, security, timeouts, safe retries, error handling, observability, and sometimes asynchronous processing.

Idempotency

Core interview question: how do you prevent duplicate side effects when a request or event is retried?

Definition

Idempotency means making the same operation once or multiple times has the same final effect. The first call may change state, but repeated calls should not create additional side effects.

DELETE /api/reservations/123

First call:  CONFIRMED -> CANCELLED
Second call: still CANCELLED

Why Retries Are Dangerous

Distributed systems retry after timeouts, dropped connections, gateway issues, service restarts, and slow downstream dependencies. The client may not know whether the server failed before processing or processed successfully but failed to return a response.

Risk: retrying a non-idempotent operation can create duplicate payments, duplicate reservations, duplicate inventory holds, duplicate loyalty points, or duplicate notifications.

HTTP Method Nuance

GET, PUT, and DELETE are usually designed to be idempotent. POST is often non-idempotent because it commonly creates a new resource or triggers a side effect. The method alone does not guarantee safety; the implementation determines the real behavior.

PUT /api/counter/increment

This is a bad design because every call increments again, even though it uses PUT.

Idempotency Key

An idempotency key is a unique value, usually generated by the client, that identifies one intended operation. If the same key is received again for the same request, the server returns the stored result instead of executing the side effect again.

POST /api/payments
Idempotency-Key: 7b9f9a3e-2e4d-4d40-9b97-9177e742ef31

The server commonly stores the idempotency key, request hash, processing status, response body, and creation time. If the same key is reused with a different request body, the server should reject it, commonly with 409 Conflict or 400 Bad Request.

Unique Constraints and Conditional Writes

A database unique constraint is stronger than a check in application code because it protects against concurrent requests. Two threads can both check for existing data at the same time and both think it does not exist. A unique constraint or conditional write is the final guard.

CREATE UNIQUE INDEX unique_reservation_deposit
ON payments (reservation_id, payment_type);

In DynamoDB, the equivalent idea is a conditional write, such as putting an item only if the idempotency key does not already exist.

Robust Processing Flow

  1. Request arrives with an idempotency key.
  2. Server tries to insert an idempotency record with status IN_PROGRESS.
  3. If insert succeeds, this request owns processing.
  4. If insert fails because the key already exists, load the existing record.
  5. If existing record is COMPLETED, return the stored response.
  6. If existing record is IN_PROGRESS, return 202 Accepted, 409 Conflict, or wait briefly depending on API design.
  7. After business processing succeeds, store the response and mark the idempotency record COMPLETED.

Event Consumers

Kafka, Kinesis, RabbitMQ, and Lambda-based consumers often operate with at-least-once delivery. That means the same event may be delivered more than once. Consumers should be idempotent by tracking event IDs, using unique constraints, using conditional writes, or making state transitions safe to repeat.

{
  "eventId": "evt-123",
  "type": "ReservationCreated",
  "reservationId": "R123"
}

Timeouts and Retries

Core interview question: how do you prevent a slow or unhealthy dependency from taking down your service?

Timeouts

A timeout is a limit on how long your service is willing to wait for a dependency. It prevents unbounded waiting and protects threads, memory, connection pools, and request latency.

Cascading failure: if Payment Service becomes slow and Reservation Service waits forever, Reservation Service threads can become stuck, queues grow, latency rises, and callers of Reservation Service may also start failing.

Connection Timeout vs Read Timeout

  • Connection timeout: how long the client waits to establish a connection to the dependency.
  • Read timeout: how long the client waits for the dependency to send a response after the connection is established.
Connection timeout: 500ms
Read timeout:       2s

Request Deadline

Timeouts should fit inside the overall request budget. If the client or gateway will give up after three seconds, a downstream timeout of ten seconds is not useful. Downstream timeouts should align with the caller's deadline and leave room for error handling.

Retries

A retry means trying an operation again after a failure. Retries help with temporary failures, but they can make outages worse if they add traffic to a service that is already unhealthy.

  • Good retry candidates: temporary timeouts, connection resets, 503, 504, and sometimes 429 while respecting Retry-After.
  • Bad retry candidates: 400, 401, 403, 404, validation errors, and business-rule failures.

Retry Safety

Before retrying, check whether the operation is safe. Reads are usually safe. Operations with side effects, such as payments, reservations, refunds, or inventory holds, need idempotency protection such as an idempotency key, unique constraint, or conditional write.

Retry Storms

A retry storm happens when many callers retry at the same time against a dependency that is already overloaded. The retry traffic increases pressure and can delay recovery.

Exponential Backoff and Jitter

Exponential backoff waits longer between each retry, such as 100ms, 200ms, and 400ms. Jitter adds randomness to those waits so many clients do not retry at exactly the same time.

Without jitter:
1000 clients retry at 100ms, then 200ms, then 400ms

With jitter:
some retry at 85ms, some at 117ms, some at 263ms

Practical Policy

A practical retry policy should limit attempts, use exponential backoff with jitter, respect the request deadline, retry only transient failures, and only retry operations that are safe to repeat.

Circuit Breakers

Core interview question: how do you stop repeated calls to a failing dependency, and how do you decide whether degraded behavior is safe?

Problem

When a dependency is slow or unavailable, repeated calls and retries can consume threads, connections, CPU, memory, and network capacity. A circuit breaker protects the caller and gives the dependency time to recover.

States

  • Closed: normal state. Calls are allowed, and the breaker records successes, failures, and slow calls.
  • Open: failure state. Calls are blocked immediately, so the service fails fast or uses a safe fallback.
  • Half-open: recovery test state. A small number of trial calls are allowed. If they succeed, the breaker closes. If they fail, it opens again.
CLOSED -> OPEN -> HALF_OPEN -> CLOSED
                   or
                   HALF_OPEN -> OPEN

Fail Fast

Fail fast means the service does not wait for a call that is likely to fail. When the circuit is open, the call is blocked immediately and returns a controlled failure or safe fallback. This keeps latency bounded and releases resources quickly.

Circuit Breaker vs Retry

Retries try again because a failure may be temporary. Circuit breakers stop trying temporarily because the dependency appears unhealthy. A common reliability pattern is timeout, limited retry, circuit breaker, then controlled failure or safe fallback.

Fallback Behavior

A fallback is an alternative response when the normal dependency call fails. Fallbacks are useful only when they are truthful and safe.

Safer Fallbacks

  • Return popular cruise activities when recommendations are unavailable.
  • Return cached non-critical content if freshness is acceptable.
  • Queue a confirmation email for retry later.
  • Accept a request as pending with 202 Accepted when asynchronous processing is appropriate.

Dangerous Fallbacks

  • Assume payment succeeded when Payment Service is unavailable.
  • Assume inventory is available when Inventory Service is unavailable.
  • Allow a request when authorization cannot be verified.
  • Return stale pricing as if it were guaranteed current pricing.

Configuration Concepts

Circuit breakers are usually configured with a failure threshold, slow-call threshold, sliding window, minimum number of calls, wait duration in open state, and permitted test calls in half-open state.

slidingWindowSize: 20
minimumNumberOfCalls: 10
failureRateThreshold: 50%
waitDurationInOpenState: 30s
permittedNumberOfCallsInHalfOpenState: 3

Observability

Circuit breaker state should be visible in metrics and logs. Useful signals include circuit state, failure rate, slow call rate, blocked calls, fallback count, dependency latency, trace ID, dependency name, and failure reason.

Error Handling and Observability

Core interview question: how do you return useful errors to clients while giving engineers enough data to debug production issues?

Good API Errors

A good API error response should be clear for the client, safe for security, stable for integrations, useful for debugging, and consistent across endpoints.

{
  "code": "PAYMENT_SERVICE_UNAVAILABLE",
  "message": "Payment service is temporarily unavailable",
  "traceId": "7f4a91c3"
}

Business Errors vs Technical Errors

  • Business errors: expected domain outcomes, such as payment declined, reservation not found, cabin unavailable, duplicate request, or reservation cannot be cancelled.
  • Technical errors: bugs, infrastructure failures, or dependency failures, such as timeouts, connection failures, database outages, null pointer exceptions, or Kafka publish failures.

Business and client errors usually map to 4xx. Unexpected server and dependency failures usually map to 5xx.

Stable Error Codes

Clients should not parse human-readable messages. They should rely on stable error codes like RESERVATION_NOT_FOUND, VALIDATION_ERROR, or PAYMENT_SERVICE_UNAVAILABLE. Messages can change; codes should remain stable.

Do Not Leak Internals

API responses should not expose stack traces, SQL queries, table names, internal hostnames, internal service URLs, raw library exceptions, secrets, or tokens. Keep internal details in logs and traces, linked by trace ID.

Spring Exception Handling

@RestControllerAdvice centralizes exception handling, keeps controllers clean, maps exceptions to appropriate status codes, and returns consistent error bodies.

@RestControllerAdvice
public class ApiExceptionHandler {
    @ExceptionHandler(ReservationNotFoundException.class)
    public ResponseEntity<ApiError> handleNotFound(ReservationNotFoundException ex) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
                .body(new ApiError(
                        "RESERVATION_NOT_FOUND",
                        "Reservation was not found",
                        getTraceId()
                ));
    }
}

Logs, Metrics, Traces

  • Logs: tell what happened in a specific execution path.
  • Metrics: show health and trends over time, such as latency, error rate, throughput, memory, consumer lag, or dependency failures.
  • Traces: show where time was spent across services in a distributed request.

Useful Log Fields

Structured logs should include fields such as timestamp, level, service, operation, trace ID, span ID, safe business identifiers, dependency name, status code, duration, error code, and exception class.

{
  "level": "ERROR",
  "service": "reservation-service",
  "operation": "createReservation",
  "traceId": "abc-123",
  "reservationId": "R123",
  "dependency": "payment-service",
  "durationMs": 2050,
  "errorCode": "PAYMENT_SERVICE_TIMEOUT",
  "exception": "ReadTimeoutException"
}

Latency Investigation

  1. Check dashboards for p95/p99 latency, error rate, throughput, and affected endpoints.
  2. Check dependency metrics for payment, inventory, database, queues, streams, and external APIs.
  3. Inspect traces for slow requests and identify the slow span.
  4. Search logs by trace ID.
  5. Check recent deploys or configuration changes.
  6. Check resource saturation: CPU, memory, thread pools, database pools, and connection pools.
  7. Choose mitigation: rollback, scale, open circuit, disable feature, queue work, or increase capacity.

Synchronous vs Asynchronous Communication

Core interview question: how do you decide whether one service should call another directly or publish an event for later processing?

Synchronous Communication

Synchronous communication means the caller waits for the callee to respond before continuing. REST and gRPC calls are common examples.

Client
  -> Reservation Service
  -> Payment Service
  -> Reservation Service
  -> Client response

Synchronous calls are useful when the caller needs an immediate answer, such as whether payment was authorized, whether a reservation exists, or whether a user is allowed to access a resource.

Synchronous Tradeoffs

  • Benefits: simple request/response model, easier to reason about, immediate result, natural fit for user-facing reads and validations.
  • Risks: tighter coupling, dependency latency affects caller latency, failures can cascade, and every dependency needs timeouts, retries, circuit breakers, and observability.

Asynchronous Communication

Asynchronous communication means the caller publishes a message or event and does not wait for the downstream work to complete inside the same request. Kafka, RabbitMQ, Kinesis, SQS, SNS, and Lambda triggers are common examples.

Reservation Service
  -> publishes ReservationCreated event
  -> returns response

Other consumers later:
  -> Email Service sends confirmation
  -> Loyalty Service awards points
  -> Analytics Service records activity

Asynchronous Tradeoffs

  • Benefits: decouples services, absorbs traffic spikes, improves resilience, supports replay, and lets non-critical work happen in the background.
  • Risks: eventual consistency, duplicate messages, ordering issues, harder debugging, schema evolution, and more complex failure handling.

When To Use Synchronous

  • The user or caller needs an immediate answer.
  • The next step depends on the result.
  • The operation is a read or direct validation.
  • The workflow cannot honestly continue without the dependency response.

Example: a reservation service may need a synchronous payment authorization before confirming a booking.

When To Use Asynchronous

  • The work can happen later.
  • The caller does not need the final result immediately.
  • The system needs to absorb high-volume events.
  • Multiple consumers need to react to the same event.
  • The system benefits from replay, buffering, or decoupling.

Example: after a reservation is confirmed, sending emails, updating analytics, awarding loyalty points, or syncing downstream reporting can happen asynchronously.

Eventual Consistency

Asynchronous systems are often eventually consistent. One service may update immediately while another service catches up later. This is acceptable when the business can tolerate short delays, but it must be visible in product behavior and support workflows.

Senior Design Rule

Use synchronous communication when correctness or user experience requires an immediate answer. Use asynchronous communication when work can be decoupled, retried, replayed, or processed in the background. Do not make everything synchronous by default, and do not make everything event-driven if the user needs a clear immediate result.

Checkpoint Notes

  • Choose synchronous communication when the caller needs the result immediately or the next step cannot proceed without it.
  • Choose asynchronous communication when the work can happen later, the user does not need the final result immediately, or multiple consumers need to react to the same event.
  • Eventual consistency means different parts of the system may temporarily show different states, but they should converge after background processing completes.
  • Async consumers need idempotency because messages may be retried or delivered more than once.
  • A Disney-style synchronous example is buying or confirming cruise tickets after checking payment and availability. A Disney-style asynchronous example is sending notifications or park news updates after an event is published.

Spring Security, OAuth2, and JWT

Core interview question: how does a Spring Boot API authenticate and authorize a request using OAuth2 and JWT?

Authentication vs Authorization

  • Authentication: verifies who the caller is.
  • Authorization: verifies what the authenticated caller is allowed to do.

A simple way to remember it: authentication is identity; authorization is permission.

OAuth2 Roles

  • Resource owner: usually the user.
  • Client: the application requesting access, such as a mobile app, web app, or backend service.
  • Authorization server: authenticates users or clients and issues tokens, such as Okta, Auth0, Cognito, Azure AD, or Ping Identity.
  • Resource server: the protected API. A Spring Boot API commonly plays this role.

JWT Structure

A JWT is commonly used as an access token and has three parts: header, payload, and signature.

header.payload.signature
  • Header: token type and signing algorithm.
  • Payload: claims such as subject, issuer, audience, expiration, scopes, and roles.
  • Signature: proves the token was signed by a trusted issuer and was not modified.

Important: a normal JWT payload is encoded, not encrypted. Do not put secrets, credentials, or sensitive data in JWT claims unless there is a specific encrypted-token design.

Resource Server Flow

  1. Request enters the Spring Security filter chain.
  2. Bearer token filter extracts the token from the Authorization header.
  3. JWT decoder validates signature and claims such as issuer, expiration, and audience.
  4. Spring creates an authenticated principal.
  5. Scopes or roles are mapped to authorities.
  6. Authorization rules are checked.
  7. The controller is called only if access is allowed.

401 vs 403

  • 401 Unauthorized: the caller is not authenticated. Common causes include missing token, expired token, invalid signature, wrong issuer, or malformed Authorization header.
  • 403 Forbidden: the caller is authenticated but not allowed. Common causes include missing scope, wrong role, or failing a business authorization rule.

Scopes, Roles, and Business Authorization

Scopes usually describe permissions, such as reservation:read or payment:authorize. Roles usually describe who the caller is, such as GUEST, ADMIN, or SERVICE_ACCOUNT.

Endpoint-level authorization checks whether the caller can access an operation. Business authorization checks whether the caller can access this specific resource.

Endpoint permission:
Can this caller use GET /api/reservations/{id}?

Business permission:
Does reservation 123 belong to this authenticated guest?

Spring Security Configuration Concept

@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    return http
            .authorizeHttpRequests(auth -> auth
                    .requestMatchers(HttpMethod.GET, "/api/reservations/**")
                    .hasAuthority("SCOPE_reservation:read")
                    .requestMatchers(HttpMethod.POST, "/api/reservations")
                    .hasAuthority("SCOPE_reservation:write")
                    .anyRequest().authenticated()
            )
            .oauth2ResourceServer(oauth2 -> oauth2.jwt())
            .build();
}

Checkpoint Notes

  • Authentication verifies identity; authorization verifies permission.
  • A Spring Boot API usually acts as an OAuth2 resource server.
  • A JWT has a header, payload, and signature.
  • Do not put secrets in a normal JWT payload because it is encoded, not encrypted.
  • 401 means missing or invalid authentication; 403 means authenticated but not permitted.
  • Endpoint-level authorization is not enough when access depends on the specific resource, such as whether a reservation belongs to the authenticated guest.

Gateway and Load Balancing

Core interview question: how does traffic flow from a client through a gateway and load balancer into backend microservices?

Common Traffic Flow

Client
  -> DNS
  -> CDN / WAF
  -> API Gateway
  -> Load Balancer
  -> Backend Service Instance
  -> Database / Cache / Event System

API Gateway

An API gateway is the front door for API traffic. It routes requests to the right backend service and often handles cross-cutting concerns such as authentication, TLS termination, rate limiting, request limits, CORS, logging, metrics, API versioning, and request or response transformation.

Load Balancer

A load balancer distributes requests across healthy instances of a service. It supports scalability, availability, fault tolerance, rolling deployments, and traffic distribution.

Gateway vs Load Balancer

Conceptually, the gateway decides which service should handle a request, while the load balancer decides which healthy instance of that service should receive it.

/api/reservations/** -> reservation-service
reservation-service -> instance 1, 2, or 3

TLS Termination

TLS termination means the HTTPS connection ends at a layer such as the gateway, load balancer, or ingress controller. Internal traffic may then continue as HTTP or be re-encrypted with TLS or mTLS depending on the security model.

Rate Limiting

Rate limiting protects services from excessive traffic, abuse, and traffic spikes. If a caller exceeds the configured limit, the API commonly returns 429 Too Many Requests.

Health Checks

A load balancer should only send traffic to healthy instances. Spring Boot Actuator commonly exposes endpoints such as /actuator/health, but health checks need careful design so non-critical dependency failures do not remove every instance from service.

  • Liveness: asks whether the process is alive or should be restarted.
  • Readiness: asks whether the instance is ready to receive traffic.

Horizontal Scaling

Horizontal scaling means adding more service instances. Backend services should usually be stateless or externalize state to a database, cache, token, or external store so any healthy instance can handle any request.

Gateway Security vs Service Security

The gateway can enforce coarse-grained security such as token validation and scopes, but services still own fine-grained business authorization. For example, the gateway may know a caller has reservation:read, while the reservation service must check whether the reservation belongs to that authenticated guest.

Checkpoint Notes

  • The gateway routes traffic to the correct service and handles API-level policies.
  • The load balancer chooses a healthy instance of that service.
  • Gateway routing chooses the service; load balancing chooses the instance.
  • Horizontally scaled services should be stateless because requests may go to different instances, and in-memory session state is not shared reliably.
  • Liveness checks whether the process should be restarted. Readiness checks whether the instance should receive traffic.
  • Gateway-level authorization is not enough for resource-specific business rules.

JUnit 5 and Mockito

Core interview question: how do you unit test Spring service behavior without starting the full application or calling real dependencies?

Unit Test vs Integration Test

  • Unit test: tests one class or small unit in isolation. It is fast and usually mocks repositories, external clients, publishers, or other boundaries.
  • Integration test: tests multiple components working together, such as controller, service, repository, Spring configuration, database mappings, or HTTP behavior.

JUnit 5

JUnit provides the test structure and assertions. Common annotations include @Test, @BeforeEach, @AfterEach, @DisplayName, and @ParameterizedTest.

@Test
void calculatesTotalPrice() {
    BigDecimal total = priceCalculator.calculate(2, new BigDecimal("50.00"));

    assertEquals(new BigDecimal("100.00"), total);
}

Mockito

Mockito creates fake versions of dependencies called mocks. Mocks let a service be tested without a real database, real API call, real network, or real message broker.

ReservationRepository repository = mock(ReservationRepository.class);
PaymentClient paymentClient = mock(PaymentClient.class);
ReservationService service = new ReservationService(repository, paymentClient);

Stubbing

Stubbing defines what a mock should return or throw when called in a test scenario.

when(repository.findById(123L))
        .thenReturn(Optional.of(reservation));

when(paymentClient.authorize(any()))
        .thenThrow(new PaymentServiceUnavailableException());

Verification

Verification checks whether a dependency was called as expected. It can also prove that something was not called when a business rule fails.

verify(repository).findById(123L);
verify(paymentClient).authorize(any(PaymentRequest.class));
verify(reservationRepository, never()).save(any());

Mockito Annotations

@ExtendWith(MockitoExtension.class)
class ReservationServiceTest {

    @Mock
    private ReservationRepository reservationRepository;

    @Mock
    private PaymentClient paymentClient;

    @InjectMocks
    private ReservationService reservationService;
}
  • @Mock creates mock dependencies.
  • @InjectMocks creates the class under test and injects the mocks.
  • @ExtendWith(MockitoExtension.class) enables Mockito in JUnit 5.

Failure Paths

Good tests cover more than the happy path. For service methods, test not found, validation or business-rule failures, declined payments, dependency failures, duplicate/idempotency cases, and cases where a downstream call must not happen.

@Test
void doesNotCancelDepartedReservation() {
    Reservation reservation = new Reservation(123L, "DEPARTED");

    when(reservationRepository.findById(123L))
            .thenReturn(Optional.of(reservation));

    assertThrows(
            ReservationCannotBeCancelledException.class,
            () -> reservationService.cancelReservation(123L)
    );

    verify(reservationRepository, never()).save(any());
}

ArgumentCaptor

ArgumentCaptor lets a test inspect the object passed to a mock, such as the entity built by a service before saving.

ArgumentCaptor<Reservation> captor = ArgumentCaptor.forClass(Reservation.class);

verify(reservationRepository).save(captor.capture());

Reservation saved = captor.getValue();
assertEquals("CONFIRMED", saved.getStatus());

Constructor Injection and Testing

Constructor injection makes unit testing easier because dependencies are explicit. A test can create the service directly with mocks and does not need to start the Spring context or rely on field injection.

Checkpoint Notes

  • A unit test checks a small unit in isolation; an integration test checks multiple components working together.
  • Mockito mocks simulate boundaries such as repositories, external APIs, publishers, and other dependencies.
  • Stubbing defines what a mock returns or throws for a scenario.
  • Verification checks whether a mock method was called, how many times, or not called at all.
  • Failure paths matter because production failures often happen in edge cases, dependency failures, and invalid state transitions.
  • Constructor injection makes tests easier because mocks can be passed directly into the class under test.

Polished Answers

What happens when a request reaches a Spring Boot service?

When a request reaches a Spring Boot service, the embedded server receives it and passes it to Spring MVC. The DispatcherServlet routes it to the matching controller method based on the HTTP method and path. The controller handles HTTP-specific concerns and delegates to a service. The service contains the business workflow and may call repositories, external services, or publish events. The repository handles persistence. The result comes back up the stack, and Spring uses Jackson to serialize the response DTO to JSON.

What is a Spring bean?

A Spring bean is an object managed by the Spring container. Spring creates it, injects its dependencies, manages its lifecycle, and can apply framework features around it. Beans can be discovered through annotations like @Service, @Repository, and @Component, or declared manually with @Bean.

Why is constructor injection preferred?

Constructor injection is preferred because dependencies are explicit, fields can be final, the object cannot be partially initialized, and the class is easy to test without Spring. It supports loose coupling and makes required collaborators visible in the class API.

What problem does Qualifier solve?

@Qualifier solves ambiguity when multiple beans match the same dependency type. If two classes implement PaymentClient, and a constructor asks for PaymentClient, Spring needs to know which implementation to inject.

Why avoid mutable request fields in services?

Most Spring beans are singleton by default, so the same service instance may handle many requests concurrently. Request-specific data should stay in method parameters or local variables. Mutable service fields can be overwritten by concurrent requests and create race conditions.

How do you design a reliable REST API?

I design REST APIs around resources, so URLs are nouns and HTTP methods express actions. I use DTOs for request and response contracts instead of exposing entities, because the API contract should be separate from the database model. I validate input at the boundary using annotations and @Valid, so invalid requests fail before business logic runs. I centralize exception handling with @RestControllerAdvice to keep controllers clean and return consistent error responses. For reliability, I think about idempotency, pagination, timeouts, retries, error codes, trace IDs, authentication, authorization, and observability. When calling another service, I make sure the call has a timeout and that retries are safe, especially for operations like payments or reservations.

What is idempotency and why does it matter?

Idempotency means that performing the same operation once or multiple times has the same final effect. It matters in distributed systems because clients, gateways, or services may retry after timeouts or network failures. If the original request succeeded but the response was lost, a retry can create duplicate side effects like duplicate payments or bookings. For risky operations, I use idempotency keys, request hashes, unique database constraints, or conditional writes. For event consumers, I assume at-least-once delivery and make handlers idempotent using event IDs or business keys.

How do you handle timeouts and retries between services?

I configure explicit timeouts for every outbound dependency call so my service does not wait indefinitely and exhaust threads, memory, or connection pools. I distinguish connection timeouts from read timeouts and align them with the overall request deadline. For retries, I only retry failures that are likely transient, such as timeouts, 503, 504, or sometimes 429 while respecting Retry-After. I avoid retrying validation, authentication, authorization, and business-rule errors. I also make sure the operation is safe to retry, either naturally idempotent or protected by an idempotency key, unique constraint, or conditional write. To avoid retry storms, I use limited attempts, exponential backoff, jitter, and eventually circuit breakers or asynchronous recovery.

How do circuit breakers and fallbacks improve reliability?

A circuit breaker prevents repeated calls to a dependency that appears unhealthy. In the closed state, calls flow normally and the breaker records successes, failures, and slow calls. If failures exceed a threshold, it opens and fails fast instead of continuing to consume resources. After a wait period, it moves to half-open and allows a small number of test calls. If they succeed, it closes; if they fail, it opens again. For fallback behavior, I only use alternatives that are truthful and safe, such as cached non-critical data, default recommendations, or queuing work for later. I would not use fallback to pretend that a payment, authorization, or inventory operation succeeded.

How do you handle errors and observability in a Spring Boot microservice?

I centralize error handling with @RestControllerAdvice so controllers stay clean and API errors are consistent. I separate expected business or client errors from unexpected server and dependency failures, map them to appropriate 4xx or 5xx status codes, and return stable error codes with safe messages and trace IDs. I avoid leaking stack traces, SQL details, tokens, or internal hostnames in API responses. For observability, I use structured logs, metrics, and distributed tracing. Logs tell me what happened, metrics show health and trends, and traces show where time was spent across services. I make sure trace or correlation IDs are propagated across service boundaries and included in logs and error responses, so production issues can be debugged quickly.

How do you choose between synchronous and asynchronous communication?

I use synchronous communication when the caller needs an immediate answer or the next step cannot proceed without the result, such as authorization, reads, validation, or payment authorization before confirming a booking. I use asynchronous communication when work can happen later, multiple consumers need to react to the same event, or the system needs buffering, replay, and decoupling. Synchronous calls are simpler but can create tight coupling and cascading failures, so they need timeouts, retries, circuit breakers, and observability. Asynchronous systems improve resilience and scalability, but require idempotent consumers, schema management, duplicate handling, ordering decisions, and acceptance of eventual consistency.

How does OAuth2/JWT work in a Spring Boot API?

In this setup, the Spring Boot API acts as an OAuth2 resource server. A client obtains an access token from an authorization server such as Okta, Cognito, or Azure AD, then sends it to the API in the Authorization: Bearer header. Spring Security's filter chain extracts the bearer token, validates the JWT signature and claims such as issuer, expiration, and audience, then creates an authenticated principal. Scopes or roles from the token are mapped to authorities and checked against endpoint rules. For finer-grained authorization, the service still needs business checks, such as whether the authenticated guest owns the reservation. If the token is missing or invalid, the API returns 401. If the token is valid but lacks permission, it returns 403.

What are the roles of a gateway and load balancer?

An API gateway is the front door for API traffic. It routes requests to the right backend service and often handles cross-cutting concerns like authentication, rate limiting, TLS termination, CORS, request limits, logging, and metrics. A load balancer distributes traffic across healthy instances of a service so the system can scale horizontally and tolerate instance failures. Conceptually, the gateway decides which service should handle the request, and the load balancer decides which instance should receive it. The gateway may enforce coarse-grained checks like token validation and scopes, but each backend service still owns fine-grained business authorization.

How do you test Spring Boot services with JUnit and Mockito?

I use JUnit 5 for test structure and assertions, and Mockito to isolate the class under test from dependencies like repositories, external clients, and publishers. For service unit tests, I usually do not start the Spring context. I instantiate the service with mocked dependencies, stub expected dependency responses, execute the method, assert the result or exception, and verify important interactions. I cover both happy paths and failure paths, such as not found, validation or business rule failures, declined payments, and dependency errors. I avoid over-mocking domain objects and prefer constructor injection because it makes unit tests simple and explicit.

Production Support

Core interview question: how do you investigate and stabilize a production issue without guessing?

Senior mindset: assess impact, stabilize customer/business impact first, then continue root cause analysis and prevention work.

Incident Flow

  1. Detect the issue through alerts, dashboards, logs, or user reports.
  2. Triage scope: affected endpoints, users, regions, versions, workflows, and time window.
  3. Mitigate if impact is high: rollback, feature flag off, scale, rate limit, open circuit, route traffic, or degrade non-critical functionality.
  4. Investigate using logs, metrics, traces, dependency dashboards, and recent change history.
  5. Resolve the root issue and verify recovery.
  6. Prevent recurrence with tests, alerts, dashboards, runbooks, guardrails, or code changes.

Golden Signals

Latency

How long requests or dependency calls take. Watch p50, p95, p99, database query time, and dependency latency.

Traffic

How much load the system receives: requests per second, transactions per minute, events per second, and consumer throughput.

Errors

Failure counts and rates: 4xx, 5xx, exceptions, timeouts, dependency failures, and publish failures.

Saturation

Resource pressure: CPU, memory, GC, thread pools, connection pools, queue depth, consumer lag, and database connections.

Latency Spike Checklist

  • Check whether one endpoint, all endpoints, one region, one version, or one customer group is affected.
  • Check traffic volume and retry behavior. Retry storms can double or triple load.
  • Check error rates and timeout counts.
  • Use traces to find the slow span: controller, service logic, downstream API, database, cache, or event publishing.
  • Check dependency dashboards: payment, inventory, database, Redis, Kafka/Kinesis, and third-party APIs.
  • Check saturation: CPU, memory, GC pauses, thread pool usage, HTTP/database connection pools.
  • Check recent deploys, configuration changes, feature flags, schema changes, and dependency releases.

Database Bottleneck Signs

Common symptoms include slow query logs, high database CPU, table scans, missing indexes, N+1 queries, lock waits, deadlocks, exhausted connection pools, long-running transactions, and query timeouts.

Important: increasing a database connection pool can make things worse if the database is already saturated. More concurrent queries can add pressure instead of improving throughput.

JVM and Thread Issues

For memory or GC issues, check heap usage, non-heap usage, allocation rate, GC pause duration, object retention, container restarts, and recent code changes. For thread pool exhaustion, check active thread count, queue size, blocked calls, missing timeouts, deadlocks, and slow downstream dependencies.

Root Cause Analysis

  • Timeline: when it started, when detected, actions taken, and when mitigated.
  • Impact: users, endpoints, regions, error rate, latency, and business workflows affected.
  • Root cause: what failed, why it failed, and why safeguards did not catch it.
  • Corrective actions: tests, monitoring, alerts, runbooks, retry/circuit policies, capacity changes, and deployment guardrails.

Modernization

Core interview question: how do you modernize a legacy Java backend without destabilizing production?

Safe Modernization Strategy

  1. Understand current behavior, dependencies, deployment process, owners, and production pain points.
  2. Add characterization tests around important existing behavior before refactoring.
  3. Upgrade incrementally: Java, Spring Boot, dependencies, build plugins, CI/CD, container images, then architecture.
  4. Keep changes small and reversible. Use feature flags, canary releases, blue/green deployments, and rollback plans.
  5. Improve observability before and during migration so regressions are visible.
  6. Document risks, compatibility changes, and operational runbooks.

Java 8/11 to 17/21 Risks

  • Removed or strongly encapsulated JDK internals and reflection behavior.
  • Dependency incompatibilities with older libraries, app servers, and build plugins.
  • Changes in GC defaults, container memory behavior, TLS/cipher support, and timezone/certificate handling.
  • Build pipeline changes: Maven compiler plugin, surefire/failsafe, Docker base images, and CI Java version.

Spring Boot Upgrade Risks

  • Spring Boot 2 to 3 includes migration from javax.* to jakarta.*.
  • Security configuration style changes, dependency version changes, and actuator endpoint changes.
  • Serialization, validation, persistence, and test configuration changes.

Strangler Pattern

The strangler pattern incrementally replaces legacy functionality with new services or modules. Route one capability at a time to the new implementation while the old system continues running. This reduces big-bang migration risk.

Interview Answer

I modernize legacy systems incrementally. First I understand behavior and production risk, then add tests and observability around critical flows. I upgrade Java, Spring Boot, dependencies, and build pipelines in small reversible steps, using feature flags, canaries, and rollback plans. For large architecture changes, I prefer a strangler approach so we can move one capability at a time instead of doing a risky big-bang rewrite.

Kinesis Deep Dive

Core interview question: how do you process high-volume real-time events with ordering, replay, and reliability?

Mental Model

Amazon Kinesis Data Streams is a managed streaming service. Producers put records into a stream. The stream is divided into shards. Consumers read records from shards and process them in near real time.

Producer -> Kinesis Stream -> Shards -> Consumers/Lambda -> Downstream systems

Key Concepts

  • Record: one event, such as ReservationCreated or GuestCheckedIn.
  • Partition key: decides which shard receives the record.
  • Shard: unit of ordering and throughput.
  • Sequence number: position of a record inside a shard.
  • Retention: how long records remain available for replay.
  • Consumer: application, Lambda, or service that reads records.

Ordering

Kinesis preserves ordering within a shard, not globally across all shards. Records with the same partition key go to the same shard, so choose partition keys based on the ordering you need.

Partition Key Design

A good partition key spreads traffic evenly while preserving required ordering. For reservation events, reservationId may preserve ordering per reservation. A bad key like eventType can create hot shards if most events share the same type.

Scaling and Hot Shards

If one shard receives too much traffic, it becomes hot and throttles writes or reads. Fixes include better partition key distribution, increasing shard count, on-demand mode where appropriate, batching, and reducing per-record size.

Consumers and Failure Handling

  • Consumers checkpoint progress after successful processing.
  • If processing fails before checkpointing, records may be read again.
  • Consumers must be idempotent because delivery is effectively at least once.
  • Poison records need a strategy: retries, skip with alert, DLQ pattern, or quarantine storage.

Kinesis vs Kafka

Kinesis is AWS-managed and shard-based, with AWS-native integrations like Lambda and CloudWatch. Kafka is broker/partition-based and often gives more ecosystem control, but requires operating a cluster unless using a managed service. Both need partition-key design, consumer lag monitoring, replay strategy, and idempotent consumers.

Interview Answer

I would use Kinesis for high-volume real-time events when AWS-native streaming, retention, replay, and Lambda integration are useful. I design partition keys around ordering and even distribution, monitor shard throughput and iterator age, and make consumers idempotent because records can be processed more than once. If lag grows, I check consumer speed, downstream dependencies, shard count, hot shards, throttling, poison messages, and batch configuration.

DynamoDB Deep Dive

Core interview question: how do you design DynamoDB tables from access patterns instead of relational habits?

Mental Model

DynamoDB is a managed NoSQL key-value/document database. It is designed for predictable low-latency access at scale, but it requires modeling around queries you need to support.

Primary Key

  • Partition key: determines where data is stored and is required for efficient lookups.
  • Sort key: orders related items within the same partition and supports range queries.
PK = GUEST#123
SK = RESERVATION#2026-06-20#R789

Access-Pattern-First Design

Start by listing queries: get reservation by id, list reservations by guest, list reservations by cruise/date, find active payment by reservation, or load itinerary events by reservation. Then design keys and indexes for those queries.

GSIs and LSIs

  • GSI: alternate partition/sort key for different query patterns. It is eventually consistent.
  • LSI: alternate sort key under the same partition key, created with the table.

Capacity and Hot Partitions

Hot partitions happen when too many reads/writes hit the same partition key. Avoid low-cardinality keys like status=ACTIVE for high-volume writes. Use high-cardinality keys and distribute load.

Conditional Writes and Idempotency

Conditional writes are critical for correctness. They can enforce put only if item does not exist, implement optimistic locking, and prevent duplicate processing with idempotency keys.

Put payment record only if paymentId does not already exist.
Put processed event only if eventId does not already exist.

TTL and Streams

TTL automatically expires items such as idempotency records or temporary sessions. DynamoDB Streams emit item-level changes, commonly consumed by Lambda for downstream processing.

Interview Answer

With DynamoDB, I start from access patterns rather than normalized relational design. I choose partition and sort keys that support the main queries and distribute load, then add GSIs for alternate lookups. I watch for hot partitions, large items, and inefficient scans. For correctness, I use conditional writes for idempotency, uniqueness, and optimistic locking. I also consider TTL for temporary data and DynamoDB Streams when downstream systems need to react to changes.

Lambda Deep Dive

Core interview question: when is Lambda appropriate, and how do you make serverless processing reliable?

Mental Model

AWS Lambda runs code in response to events without managing servers. It is a good fit for event processing, glue logic, scheduled tasks, lightweight APIs, stream processing, and background jobs.

Common Event Sources

  • API Gateway for HTTP APIs.
  • Kinesis for streaming records.
  • DynamoDB Streams for table change events.
  • SQS for queued work and buffering.
  • S3 for object-created events.
  • EventBridge for scheduled or business events.

Cold Starts

A cold start happens when AWS initializes a new execution environment before invoking the function. It can add latency, especially with large packages, VPC networking, heavy frameworks, or low traffic. Mitigations include smaller packages, lighter initialization, provisioned concurrency for critical paths, and avoiding unnecessary startup work.

Concurrency

Lambda scales by running concurrent executions. Reserved concurrency can protect downstream systems and isolate critical functions. Too much concurrency can overwhelm databases, APIs, or streams.

Retries and DLQs

Retry behavior depends on the event source. Async invokes may retry automatically. Streams retry batches until success or until records expire unless configured with partial batch handling and failure destinations. SQS can use visibility timeout and dead-letter queues.

Idempotency

Lambda handlers must be idempotent because events can be retried. Use event IDs, idempotency tables, conditional writes, and safe state transitions to avoid duplicate side effects.

When Not To Use Lambda

  • Long-running workloads beyond Lambda limits.
  • Very latency-sensitive APIs where cold starts are unacceptable without mitigation.
  • Complex stateful workflows better handled by services or Step Functions.
  • Heavy continuous compute where containers may be more cost-effective.

Interview Answer

I use Lambda when event-driven, bursty, or background processing fits the problem, such as Kinesis consumers, DynamoDB Stream handlers, scheduled jobs, and lightweight APIs. I design handlers to be idempotent, monitor errors and duration, control concurrency to protect downstream systems, and understand retry behavior for each event source. For critical latency paths, I consider cold starts, package size, initialization work, and provisioned concurrency.

Event-Driven Architecture

Core interview question: how do you design an event-driven backend that handles real-time data safely?

Core Ideas

  • Event: a fact that happened, such as ReservationCreated.
  • Command: a request to do something, such as CreateReservation.
  • Producer: publishes events.
  • Consumer: reacts to events.
  • Broker/stream: Kafka, RabbitMQ, Kinesis, SNS/SQS, or EventBridge.

Reliability Rules

  • Assume at-least-once delivery and make consumers idempotent.
  • Use stable event IDs and business keys.
  • Design schemas carefully and evolve them backward-compatibly.
  • Have a retry and poison-message strategy.
  • Monitor lag, error rate, processing time, DLQ count, and downstream dependency health.

Outbox Pattern

The outbox pattern avoids losing events when database updates and event publishing must happen together. Save the business entity and an outbox event in the same database transaction. A separate worker publishes the event and marks it as published.

1. Save reservation and outbox row in one transaction.
2. Worker reads unpublished outbox rows.
3. Worker publishes ReservationCreated.
4. Worker marks outbox row as published.

Saga Pattern

A saga coordinates a multi-step distributed workflow without a single database transaction. Each step has a compensating action if later steps fail. Example: reserve cabin, authorize payment, confirm booking. If payment fails, release the cabin hold.

Kafka vs RabbitMQ vs Kinesis

  • Kafka: durable log, partitions, replay, high-throughput event streaming.
  • RabbitMQ: queue/message routing, work queues, flexible routing patterns.
  • Kinesis: AWS-managed stream, shards, retention, replay, Lambda integration.

Interview Answer

For event-driven systems, I treat events as facts and design consumers for at-least-once delivery. That means stable event IDs, idempotent handlers, duplicate detection, retry limits, poison-message handling, and observability around lag and errors. I use the outbox pattern when a database change and event publish must stay consistent. I also design event schemas to evolve safely and choose Kafka, RabbitMQ, or Kinesis based on throughput, replay, routing, cloud integration, and operational needs.
