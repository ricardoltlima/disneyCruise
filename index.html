Spring Boot Request Lifecycle

Core interview question: what happens when an HTTP request reaches a Spring Boot backend application?

Client sends GET /api/reservations/123 Load balancer or gateway forwards traffic to the service Embedded server, usually Tomcat, receives the HTTP request Spring MVC DispatcherServlet receives the request Spring maps the request to the correct controller method Controller handles HTTP boundary concerns Service layer executes the business workflow Repository or client accesses data or external systems Response DTO returns through the controller Jackson serializes Java object to JSON

Spring Boot

Spring Boot helps build production-ready Java services by providing auto-configuration, embedded servers, dependency injection, and integrations for web APIs, persistence, security, monitoring, and testing.

Embedded Server

A Spring Boot REST API usually runs with an embedded server such as Tomcat. Instead of deploying a WAR file to an external server, the application can often be started with:

java -jar reservation-service.jar

DispatcherServlet

The DispatcherServlet is the central entry point for Spring MVC requests. It receives the HTTP request, uses handler mappings to find the matching controller method, invokes that method, and coordinates the response.

Controller

The controller owns the HTTP boundary: routes, path variables, query parameters, request bodies, validation trigger, and response status. Controllers should stay thin and delegate business rules to services.

@RestController
@RequestMapping("/api/reservations")
public class ReservationController {
    private final ReservationService reservationService;

    public ReservationController(ReservationService reservationService) {
        this.reservationService = reservationService;
    }

    @GetMapping("/{id}")
    public ReservationResponse getReservation(@PathVariable Long id) {
        return reservationService.getReservation(id);
    }
}

Service

The service layer owns the business workflow. It applies business rules, coordinates repositories and external clients, manages transactions, handles domain-level exceptions, and decides what should happen for a use case.

@Service
public class ReservationService {
    private final ReservationRepository reservationRepository;

    public ReservationService(ReservationRepository reservationRepository) {
        this.reservationRepository = reservationRepository;
    }

    public ReservationResponse getReservation(Long id) {
        Reservation reservation = reservationRepository.findById(id)
                .orElseThrow(() -> new ReservationNotFoundException(id));

        return new ReservationResponse(
                reservation.getId(),
                reservation.getGuestName(),
                reservation.getStatus()
        );
    }
}

Repository

The repository layer abstracts persistence. With Spring Data JPA, the framework can generate implementations for common operations such as findById, save, delete, and findAll.

@Repository
public interface ReservationRepository extends JpaRepository<Reservation, Long> {
}

DTO vs Entity

DTOs define the shape of data crossing the API boundary. Entities represent persisted domain data and are usually mapped to database tables or documents. Avoid exposing entities directly from APIs because that can leak internal database structure and make API contracts harder to evolve.

public record ReservationResponse(
        Long id,
        String guestName,
        String status
) {}

Beans and Dependency Injection

Core interview question: how does Spring create objects and connect them together?

Spring Bean

A Spring bean is an object managed by the Spring container. Spring creates it, injects its dependencies, manages its lifecycle, and can apply framework behavior around it.

Beans can be discovered with annotations such as @Component, @Service, @Repository, @Controller, @RestController, and @Configuration. They can also be declared manually through @Bean methods.

Spring Container

The Spring container, usually represented by ApplicationContext, discovers bean definitions, creates objects, wires dependencies, manages lifecycle callbacks, and makes beans available for injection.

@SpringBootApplication
public class ReservationApplication {
    public static void main(String[] args) {
        SpringApplication.run(ReservationApplication.class, args);
    }
}

Component Scanning

@SpringBootApplication includes component scanning. By default, Spring scans the package where the main application class lives and its subpackages. If a bean is not being injected, one thing to check is whether it is inside the component scan path.

Constructor Injection

Constructor injection is preferred because required dependencies are explicit, fields can be final, objects cannot be created in an incomplete state, and classes are easier to unit test without starting Spring.

@Service
public class ReservationService {
    private final ReservationRepository reservationRepository;

    public ReservationService(ReservationRepository reservationRepository) {
        this.reservationRepository = reservationRepository;
    }
}

Multiple Beans and Qualifier

If multiple beans match the same dependency type, Spring needs disambiguation. Use @Qualifier to name the dependency explicitly or @Primary to define a default bean.

public PaymentService(@Qualifier("stripePaymentClient") PaymentClient paymentClient) {
    this.paymentClient = paymentClient;
}

Bean Scope and Concurrency

Most Spring beans are singleton by default, meaning one instance per Spring container. Since the same service instance can handle many requests concurrently, request-specific mutable data should stay in method parameters or local variables, not service fields.

Avoid this pattern: storing currentReservationId or similar request data as a field in a singleton service. Concurrent requests can overwrite each other's data and cause race conditions.

Configuration Beans

Use @Configuration and @Bean when you need custom construction logic or want to register a third-party object with Spring.

@Configuration
public class AwsConfig {
    @Bean
    public DynamoDbClient dynamoDbClient() {
        return DynamoDbClient.builder()
                .region(Region.US_EAST_1)
                .build();
    }
}

Bean Lifecycle

  1. Bean definition discovered.
  2. Object instantiated.
  3. Dependencies injected.
  4. Initialization callbacks run.
  5. Bean is used by the application.
  6. Shutdown callbacks run when the app stops.

REST API Design

Core interview question: how do you design a clean, reliable REST API in Spring Boot?

Resource-Oriented URLs

REST APIs usually model URLs around resources, not actions. The URL identifies the thing, and the HTTP method expresses the action.

Good

GET /api/reservations/123
POST /api/reservations
PATCH /api/reservations/123
DELETE /api/reservations/123

Less Ideal

GET /api/getReservation?id=123
POST /api/createReservation
POST /api/updateReservation
POST /api/deleteReservation

HTTP Methods

  • GET: read a resource.
  • POST: create a resource or trigger a non-idempotent operation.
  • PUT: replace a resource.
  • PATCH: partially update a resource.
  • DELETE: delete or cancel a resource.

Status Codes

  • 200 OK: successful read or update.
  • 201 Created: resource created, often with a Location header.
  • 202 Accepted: request accepted for asynchronous processing.
  • 204 No Content: successful operation with no response body.
  • 400 Bad Request: invalid request syntax or validation failure.
  • 401 Unauthorized: missing or invalid authentication.
  • 403 Forbidden: authenticated caller is not allowed.
  • 404 Not Found: resource does not exist.
  • 409 Conflict: state conflict or duplicate business operation.
  • 429 Too Many Requests: rate limit exceeded.
  • 500 Internal Server Error: unexpected server failure.
  • 503 Service Unavailable: temporary outage or dependency unavailable.

DTOs

Use DTOs for request and response contracts instead of exposing entities directly. DTOs separate the API contract from the persistence model, prevent leaking internal fields or sensitive data, and let the API evolve without forcing every database change to become a client-facing change.

public record CreateReservationRequest(
        @NotBlank String guestName,
        @NotNull Long cruiseId,
        @Min(1) int partySize
) {}

public record ReservationResponse(
        Long id,
        String guestName,
        Long cruiseId,
        int partySize,
        String status
) {}

Validation

@Valid tells Spring to apply validation rules on the request DTO before the controller method continues. If validation fails, Spring raises a validation exception and the service layer is not called.

@PostMapping
public ResponseEntity<ReservationResponse> createReservation(
        @Valid @RequestBody CreateReservationRequest request
) {
    ReservationResponse response = reservationService.createReservation(request);
    URI location = URI.create("/api/reservations/" + response.id());
    return ResponseEntity.created(location).body(response);
}

Global Exception Handling

@RestControllerAdvice centralizes exception handling across controllers. It keeps controllers clean, maps exceptions to proper HTTP status codes, and returns consistent error responses.

@RestControllerAdvice
public class ApiExceptionHandler {
    @ExceptionHandler(ReservationNotFoundException.class)
    public ResponseEntity<ApiError> handleNotFound(ReservationNotFoundException ex) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
                .body(new ApiError("RESERVATION_NOT_FOUND", ex.getMessage()));
    }
}

Error Response Design

Reliable APIs should return predictable errors with stable error codes, clear messages, and trace IDs for supportability.

{
  "code": "VALIDATION_ERROR",
  "message": "Request validation failed",
  "traceId": "abc-123",
  "fieldErrors": [
    {
      "field": "partySize",
      "message": "partySize must be at least 1"
    }
  ]
}

Pagination and Versioning

Avoid unbounded list endpoints. Use page and size for simple cases, and cursor-based pagination for high-volume or frequently changing datasets.

GET /api/reservations?page=0&size=50
GET /api/reservations?limit=50&cursor=eyJpZCI6MTIzfQ

Prefer backward-compatible API changes when possible. For breaking changes, use a clear versioning strategy such as /api/v1/reservations and a migration plan.

Idempotency

Core interview question: how do you prevent duplicate side effects when a request or event is retried?

Definition

Idempotency means making the same operation once or multiple times has the same final effect. The first call may change state, but repeated calls should not create additional side effects.

DELETE /api/reservations/123

First call:  CONFIRMED -> CANCELLED
Second call: still CANCELLED

Why Retries Are Dangerous

Distributed systems retry after timeouts, dropped connections, gateway issues, service restarts, and slow downstream dependencies. The client may not know whether the server failed before processing or processed successfully but failed to return a response.

Risk: retrying a non-idempotent operation can create duplicate payments, duplicate reservations, duplicate inventory holds, duplicate loyalty points, or duplicate notifications.

HTTP Method Nuance

GET, PUT, and DELETE are usually designed to be idempotent. POST is often non-idempotent because it commonly creates a new resource or triggers a side effect. The method alone does not guarantee safety; the implementation determines the real behavior.

PUT /api/counter/increment

This is a bad design because every call increments again, even though it uses PUT.

Idempotency Key

An idempotency key is a unique value, usually generated by the client, that identifies one intended operation. If the same key is received again for the same request, the server returns the stored result instead of executing the side effect again.

POST /api/payments
Idempotency-Key: 7b9f9a3e-2e4d-4d40-9b97-9177e742ef31

The server commonly stores the idempotency key, request hash, processing status, response body, and creation time. If the same key is reused with a different request body, the server should reject it, commonly with 409 Conflict or 400 Bad Request.

Unique Constraints and Conditional Writes

A database unique constraint is stronger than a check in application code because it protects against concurrent requests. Two threads can both check for existing data at the same time and both think it does not exist. A unique constraint or conditional write is the final guard.

CREATE UNIQUE INDEX unique_reservation_deposit
ON payments (reservation_id, payment_type);

In DynamoDB, the equivalent idea is a conditional write, such as putting an item only if the idempotency key does not already exist.

Robust Processing Flow

  1. Request arrives with an idempotency key.
  2. Server tries to insert an idempotency record with status IN_PROGRESS.
  3. If insert succeeds, this request owns processing.
  4. If insert fails because the key already exists, load the existing record.
  5. If existing record is COMPLETED, return the stored response.
  6. If existing record is IN_PROGRESS, return 202 Accepted, 409 Conflict, or wait briefly depending on API design.
  7. After business processing succeeds, store the response and mark the idempotency record COMPLETED.

Event Consumers

Kafka, Kinesis, RabbitMQ, and Lambda-based consumers often operate with at-least-once delivery. That means the same event may be delivered more than once. Consumers should be idempotent by tracking event IDs, using unique constraints, using conditional writes, or making state transitions safe to repeat.

{
  "eventId": "evt-123",
  "type": "ReservationCreated",
  "reservationId": "R123"
}

Timeouts and Retries

Core interview question: how do you prevent a slow or unhealthy dependency from taking down your service?

Timeouts

A timeout is a limit on how long your service is willing to wait for a dependency. It prevents unbounded waiting and protects threads, memory, connection pools, and request latency.

Cascading failure: if Payment Service becomes slow and Reservation Service waits forever, Reservation Service threads can become stuck, queues grow, latency rises, and callers of Reservation Service may also start failing.

Connection Timeout vs Read Timeout

  • Connection timeout: how long the client waits to establish a connection to the dependency.
  • Read timeout: how long the client waits for the dependency to send a response after the connection is established.
Connection timeout: 500ms
Read timeout:       2s

Request Deadline

Timeouts should fit inside the overall request budget. If the client or gateway will give up after three seconds, a downstream timeout of ten seconds is not useful. Downstream timeouts should align with the caller's deadline and leave room for error handling.

Retries

A retry means trying an operation again after a failure. Retries help with temporary failures, but they can make outages worse if they add traffic to a service that is already unhealthy.

  • Good retry candidates: temporary timeouts, connection resets, 503, 504, and sometimes 429 while respecting Retry-After.
  • Bad retry candidates: 400, 401, 403, 404, validation errors, and business-rule failures.

Retry Safety

Before retrying, check whether the operation is safe. Reads are usually safe. Operations with side effects, such as payments, reservations, refunds, or inventory holds, need idempotency protection such as an idempotency key, unique constraint, or conditional write.

Retry Storms

A retry storm happens when many callers retry at the same time against a dependency that is already overloaded. The retry traffic increases pressure and can delay recovery.

Exponential Backoff and Jitter

Exponential backoff waits longer between each retry, such as 100ms, 200ms, and 400ms. Jitter adds randomness to those waits so many clients do not retry at exactly the same time.

Without jitter:
1000 clients retry at 100ms, then 200ms, then 400ms

With jitter:
some retry at 85ms, some at 117ms, some at 263ms

Practical Policy

A practical retry policy should limit attempts, use exponential backoff with jitter, respect the request deadline, retry only transient failures, and only retry operations that are safe to repeat.

Circuit Breakers

Core interview question: how do you stop repeated calls to a failing dependency, and how do you decide whether degraded behavior is safe?

Problem

When a dependency is slow or unavailable, repeated calls and retries can consume threads, connections, CPU, memory, and network capacity. A circuit breaker protects the caller and gives the dependency time to recover.

States

  • Closed: normal state. Calls are allowed, and the breaker records successes, failures, and slow calls.
  • Open: failure state. Calls are blocked immediately, so the service fails fast or uses a safe fallback.
  • Half-open: recovery test state. A small number of trial calls are allowed. If they succeed, the breaker closes. If they fail, it opens again.
CLOSED -> OPEN -> HALF_OPEN -> CLOSED
                   or
                   HALF_OPEN -> OPEN

Fail Fast

Fail fast means the service does not wait for a call that is likely to fail. When the circuit is open, the call is blocked immediately and returns a controlled failure or safe fallback. This keeps latency bounded and releases resources quickly.

Circuit Breaker vs Retry

Retries try again because a failure may be temporary. Circuit breakers stop trying temporarily because the dependency appears unhealthy. A common reliability pattern is timeout, limited retry, circuit breaker, then controlled failure or safe fallback.

Fallback Behavior

A fallback is an alternative response when the normal dependency call fails. Fallbacks are useful only when they are truthful and safe.

Safer Fallbacks

  • Return popular cruise activities when recommendations are unavailable.
  • Return cached non-critical content if freshness is acceptable.
  • Queue a confirmation email for retry later.
  • Accept a request as pending with 202 Accepted when asynchronous processing is appropriate.

Dangerous Fallbacks

  • Assume payment succeeded when Payment Service is unavailable.
  • Assume inventory is available when Inventory Service is unavailable.
  • Allow a request when authorization cannot be verified.
  • Return stale pricing as if it were guaranteed current pricing.

Configuration Concepts

Circuit breakers are usually configured with a failure threshold, slow-call threshold, sliding window, minimum number of calls, wait duration in open state, and permitted test calls in half-open state.

slidingWindowSize: 20
minimumNumberOfCalls: 10
failureRateThreshold: 50%
waitDurationInOpenState: 30s
permittedNumberOfCallsInHalfOpenState: 3

Observability

Circuit breaker state should be visible in metrics and logs. Useful signals include circuit state, failure rate, slow call rate, blocked calls, fallback count, dependency latency, trace ID, dependency name, and failure reason.

Error Handling and Observability

Core interview question: how do you return useful errors to clients while giving engineers enough data to debug production issues?

Good API Errors

A good API error response should be clear for the client, safe for security, stable for integrations, useful for debugging, and consistent across endpoints.

{
  "code": "PAYMENT_SERVICE_UNAVAILABLE",
  "message": "Payment service is temporarily unavailable",
  "traceId": "7f4a91c3"
}

Business Errors vs Technical Errors

  • Business errors: expected domain outcomes, such as payment declined, reservation not found, cabin unavailable, duplicate request, or reservation cannot be cancelled.
  • Technical errors: bugs, infrastructure failures, or dependency failures, such as timeouts, connection failures, database outages, null pointer exceptions, or Kafka publish failures.

Business and client errors usually map to 4xx. Unexpected server and dependency failures usually map to 5xx.

Stable Error Codes

Clients should not parse human-readable messages. They should rely on stable error codes like RESERVATION_NOT_FOUND, VALIDATION_ERROR, or PAYMENT_SERVICE_UNAVAILABLE. Messages can change; codes should remain stable.

Do Not Leak Internals

API responses should not expose stack traces, SQL queries, table names, internal hostnames, internal service URLs, raw library exceptions, secrets, or tokens. Keep internal details in logs and traces, linked by trace ID.

Spring Exception Handling

@RestControllerAdvice centralizes exception handling, keeps controllers clean, maps exceptions to appropriate status codes, and returns consistent error bodies.

@RestControllerAdvice
public class ApiExceptionHandler {
    @ExceptionHandler(ReservationNotFoundException.class)
    public ResponseEntity<ApiError> handleNotFound(ReservationNotFoundException ex) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
                .body(new ApiError(
                        "RESERVATION_NOT_FOUND",
                        "Reservation was not found",
                        getTraceId()
                ));
    }
}

Logs, Metrics, Traces

  • Logs: tell what happened in a specific execution path.
  • Metrics: show health and trends over time, such as latency, error rate, throughput, memory, consumer lag, or dependency failures.
  • Traces: show where time was spent across services in a distributed request.

Useful Log Fields

Structured logs should include fields such as timestamp, level, service, operation, trace ID, span ID, safe business identifiers, dependency name, status code, duration, error code, and exception class.

{
  "level": "ERROR",
  "service": "reservation-service",
  "operation": "createReservation",
  "traceId": "abc-123",
  "reservationId": "R123",
  "dependency": "payment-service",
  "durationMs": 2050,
  "errorCode": "PAYMENT_SERVICE_TIMEOUT",
  "exception": "ReadTimeoutException"
}

Latency Investigation

  1. Check dashboards for p95/p99 latency, error rate, throughput, and affected endpoints.
  2. Check dependency metrics for payment, inventory, database, queues, streams, and external APIs.
  3. Inspect traces for slow requests and identify the slow span.
  4. Search logs by trace ID.
  5. Check recent deploys or configuration changes.
  6. Check resource saturation: CPU, memory, thread pools, database pools, and connection pools.
  7. Choose mitigation: rollback, scale, open circuit, disable feature, queue work, or increase capacity.

Polished Answers

What happens when a request reaches a Spring Boot service?

When a request reaches a Spring Boot service, the embedded server receives it and passes it to Spring MVC. The DispatcherServlet routes it to the matching controller method based on the HTTP method and path. The controller handles HTTP-specific concerns and delegates to a service. The service contains the business workflow and may call repositories, external services, or publish events. The repository handles persistence. The result comes back up the stack, and Spring uses Jackson to serialize the response DTO to JSON.

What is a Spring bean?

A Spring bean is an object managed by the Spring container. Spring creates it, injects its dependencies, manages its lifecycle, and can apply framework features around it. Beans can be discovered through annotations like @Service, @Repository, and @Component, or declared manually with @Bean.

Why is constructor injection preferred?

Constructor injection is preferred because dependencies are explicit, fields can be final, the object cannot be partially initialized, and the class is easy to test without Spring. It supports loose coupling and makes required collaborators visible in the class API.

What problem does Qualifier solve?

@Qualifier solves ambiguity when multiple beans match the same dependency type. If two classes implement PaymentClient, and a constructor asks for PaymentClient, Spring needs to know which implementation to inject.

Why avoid mutable request fields in services?

Most Spring beans are singleton by default, so the same service instance may handle many requests concurrently. Request-specific data should stay in method parameters or local variables. Mutable service fields can be overwritten by concurrent requests and create race conditions.

How do you design a reliable REST API?

I design REST APIs around resources, so URLs are nouns and HTTP methods express actions. I use DTOs for request and response contracts instead of exposing entities, because the API contract should be separate from the database model. I validate input at the boundary using annotations and @Valid, so invalid requests fail before business logic runs. I centralize exception handling with @RestControllerAdvice to keep controllers clean and return consistent error responses. For reliability, I think about idempotency, pagination, timeouts, retries, error codes, trace IDs, authentication, authorization, and observability. When calling another service, I make sure the call has a timeout and that retries are safe, especially for operations like payments or reservations.

What is idempotency and why does it matter?

Idempotency means that performing the same operation once or multiple times has the same final effect. It matters in distributed systems because clients, gateways, or services may retry after timeouts or network failures. If the original request succeeded but the response was lost, a retry can create duplicate side effects like duplicate payments or bookings. For risky operations, I use idempotency keys, request hashes, unique database constraints, or conditional writes. For event consumers, I assume at-least-once delivery and make handlers idempotent using event IDs or business keys.

How do you handle timeouts and retries between services?

I configure explicit timeouts for every outbound dependency call so my service does not wait indefinitely and exhaust threads, memory, or connection pools. I distinguish connection timeouts from read timeouts and align them with the overall request deadline. For retries, I only retry failures that are likely transient, such as timeouts, 503, 504, or sometimes 429 while respecting Retry-After. I avoid retrying validation, authentication, authorization, and business-rule errors. I also make sure the operation is safe to retry, either naturally idempotent or protected by an idempotency key, unique constraint, or conditional write. To avoid retry storms, I use limited attempts, exponential backoff, jitter, and eventually circuit breakers or asynchronous recovery.

How do circuit breakers and fallbacks improve reliability?

A circuit breaker prevents repeated calls to a dependency that appears unhealthy. In the closed state, calls flow normally and the breaker records successes, failures, and slow calls. If failures exceed a threshold, it opens and fails fast instead of continuing to consume resources. After a wait period, it moves to half-open and allows a small number of test calls. If they succeed, it closes; if they fail, it opens again. For fallback behavior, I only use alternatives that are truthful and safe, such as cached non-critical data, default recommendations, or queuing work for later. I would not use fallback to pretend that a payment, authorization, or inventory operation succeeded.

How do you handle errors and observability in a Spring Boot microservice?

I centralize error handling with @RestControllerAdvice so controllers stay clean and API errors are consistent. I separate expected business or client errors from unexpected server and dependency failures, map them to appropriate 4xx or 5xx status codes, and return stable error codes with safe messages and trace IDs. I avoid leaking stack traces, SQL details, tokens, or internal hostnames in API responses. For observability, I use structured logs, metrics, and distributed tracing. Logs tell me what happened, metrics show health and trends, and traces show where time was spent across services. I make sure trace or correlation IDs are propagated across service boundaries and included in logs and error responses, so production issues can be debugged quickly.

AI at Work

For an interview, AI usage should sound like senior engineering judgment, not blind tool adoption.

Strong framing: I use AI to accelerate understanding, boilerplate, tests, refactoring ideas, and documentation, but I keep ownership of correctness, security, architecture, and production behavior.

Good Uses

  • Understanding unfamiliar code.
  • Generating first drafts of unit tests.
  • Explaining stack traces.
  • Summarizing pull requests.
  • Suggesting refactoring approaches.
  • Creating migration checklists.

What To Avoid

  • Pasting secrets, credentials, tokens, customer data, or unapproved proprietary information into AI tools.
  • Accepting generated code without review.
  • Using AI-generated security or authorization code blindly.
  • Letting AI perform destructive actions without approval.
  • Relying on AI for production diagnosis without validating logs, metrics, and traces.
  • Introducing dependencies or licenses without checking policy.

References to know: NIST AI Risk Management Framework, NIST Generative AI Profile, OWASP Top 10 for LLM Applications, and GitHub Copilot Trust Center.
